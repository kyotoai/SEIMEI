Introduction

The main README describes SEIMEI as the Search-Enhanced Interface for Multi-Expertise Integration—KyotoAI's agent system that lets thousands of reasoning specialists collaborate through a reward-model search loop. Instead of retraining the base LLM, SEIMEI tunes a lightweight reranker (RMSearch) to repeatedly pick the next document and agent trajectory, keeping chains-of-thought grounded.

  • Search the best agent. The README example shows how each agent reads documents, interacts with the LLM, and feeds an answer back through the search engine so only promising threads survive.
  • Trainable reranker. The proprietary reward model outperforms embedding-only retrieval because it learns which agent solves a question, not just which text looks similar.
  • Applied workflows. Screenshots in README.md highlight scientific research, software debugging, and experimentation pipelines built on SEIMEI.

Quick Start

Everything below is lifted from the “Getting Started” and “Installation” guidance in README.md together with the quick start block inside seimei/README.md.

  1. Clone the repository and install it in editable mode so local changes propagate immediately.
  2. Export your OPENAI_API_KEY and KYOTOAI_API_KEY.
  3. Run seimei from the CLI or use the orchestrator in Python.
git clone https://github.com/kyotoai/SEIMEI.git
pip install -e SEIMEI/
seimei
            
export OPENAI_API_KEY="(your_openai_api_key)"
export KYOTOAI_API_KEY="(your_kyotoai_api_key)"

Code Act

Section 6 of seimei/README.md lists the bundled agents; agents/code_act.py executes whitelisted shell commands so long-running experiments stay grounded. Enable it with allow_code_exec=True and optionally restrict allowed_commands (README suggests ["python", "pip", "ls", "cat"] as a starting point).

The Python block below is copied verbatim from the README Quick start example and demonstrates how seimei drives code_act alongside RMSearch.

Python 
import asyncio
from seimei import seimei  # class name is `seimei` (lowercase) for convenience

async def demo_code_act():
    orchestrator = seimei(
        llm_kwargs={"model": "gpt-5-nano"},
        rm_kwargs={"url": "https://kyotoai.net/v1/rmsearch", "agent_routing":False, "knowledge_search":True},
        allow_code_exec=True,
        agent_log_head_lines=1,
        max_tokens_per_question=30000,
        load_knowledge_path="seimei_knowledge/knwoledge.csv",
    )

    result = await orchestrator(
        messages=[
            {"role": "user", "content": "Design a single 7-day endgame plan for my turbulence surrogate project based on my past history."},
        ],
    )

asyncio.run(demo_code_act())

Custom Agent

The README's “Create an agent” walkthrough shows how to add a bespoke planner to the registry by subclassing Agent and reusing the shared search helper. Drop the file into a folder pointed to by agent_config and SEIMEI registers it automatically.

Python 
from seimei import Agent

class prioritise_docs(Agent):
    """Rank documentation chunks that should be read next."""

    description = "Select document chunks relevant to the latest user request."

    async def inference(self, messages, shared_ctx, **kwargs):
        search = shared_ctx.get("search")
        if not search:
            return {"content": "search helper unavailable", "log": {}}

        question = next((m["content"] for m in reversed(messages) if m.get("role") == "user"), "")
        candidates = [{"key": text, "section": name} for name, text in kwargs.get("docs", [])]

        ranked = await search(
            query=question,
            keys=candidates,
            k=3,
            context={"purpose": "doc_ranking"},
        )
        plan = "\n".join(f"- {item['payload']['section']}" for item in ranked if item.get("payload"))
        return {"content": f"Review next:\n{plan}", "log": {"query": question, "sections": ranked}}

Each agent receives messages, the shared_ctx (search helper, instruction list, LLM proxy), and any keyword args you pass through agent_config. The README also calls out how to override instruction_list when you want to re-steer the planner.

Dataset Logging

Pulling from the “Dataset logging” block in seimei/README.md: every run writes structured artifacts into log_dir, including full message history, agent steps, and metadata. That makes it trivial to replay or finetune the reward model. 

log_dir/
  dataset.jsonl        # one JSON per run (append-only)
  run-YYYYmmdd-HHMMSS-<uuid>/
    messages.json      # full message history
    steps.jsonl        # one line per agent step
    output.txt         # final assistant text
    meta.json          # run metadata (model, timings, etc.)

Each JSON record exposes schema_version, run_id, input_messages, and aggregated token usage. If a token cap is hit, the limiter snapshot is persisted so you can diagnose safety stops later on.

Rate Limits

Throughput tiers

Tier A – shared lab

~20 concurrent agent turns, up to 200k tokens/minute aggregate.

Tier B – managed pilot

~80 concurrent agent turns, ~600k tokens/minute aggregate.

Tier C – dedicated cluster

Single-tenant deployments with bespoke RMSearch models and SLAs negotiated per project.

Burst mode

Available once logs show steady-state accuracy <2% variance and tooling covers rollback or throttling hooks.

When you exceed the soft limit, requests may still succeed but be throttled into a higher-latency mode. For production deployments we recommend aggregating queries and using streaming where possible.
